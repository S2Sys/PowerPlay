## 🔐 Zero-Trust Security

**Core Principle**: Never trust by default. Verify every identity, encrypt everything, assume breach.

### Key Controls

#### 1. Identity & Access Management
- **MFA mandatory** for all users (OAuth 2.0, PKCE, JWT tokens)
- **Short-lived tokens** (1-24 hours, not permanent API keys)
- **Conditional access** (device health, location, risk)
- **Just-in-time elevation** (temporary elevated access)

**Example**:
```csharp
// OAuth 2.0 + PKCE (most secure for browser apps)
var tokenRequest = new
{
    grant_type = "authorization_code",
    code = authCode,
    code_verifier = codeVerifier,  // PKCE
    client_id = clientId
};

// Short-lived token (1 hour)
var response = await _auth.GetTokenAsync(tokenRequest);
var token = response.AccessToken;  // Expires in 1 hour

// JIT elevation (require re-auth for sensitive ops)
if (action == "DeleteUser")
    await RequireMFAAsync(userId);  // Challenge again
```

#### 2. Authentication
- **Verify every identity** (never trust IP/location)
- **Multi-factor authentication** (authenticator app, hardware key, SMS fallback)
- **Continuous validation** (re-check credentials periodically)

#### 3. Authorization
- **Least privilege** (minimum permissions needed)
- **Role-based access control (RBAC)** (reader, writer, admin)
- **Attribute-based access control (ABAC)** (fine-grained rules)
- **Audit privilege escalations** (who escalated, when, why)

**Example**:
```yaml
# RBAC - Limit what each role can do
roles:
  reader:
    permissions:
      - orders:read
      - orders:list
  
  writer:
    permissions:
      - orders:read
      - orders:write
      - orders:delete  # Own orders only
  
  admin:
    permissions:
      - "*"  # Full access (rare)
```

#### 4. Network Segmentation
- **Microsegmentation** (divide network into zones)
- **No flat networks** (assume all servers can reach all)
- **Service-to-service mTLS** (mutual TLS between services)
- **Network policies** (deny by default, allow specific)

**Example**:
```yaml
# Kubernetes Network Policy - Allow only trusted traffic
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-policy
spec:
  podSelector:
    matchLabels:
      app: api
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: ingress-controller  # Only from ingress
    ports:
    - protocol: TCP
      port: 3000
```

#### 5. Encryption
- **TLS 1.2+** for all data in transit (HTTPS, mTLS)
- **AES-256** for data at rest
- **Encrypt backups** (same strength as production)
- **Key management** (HSM, Key Vault, rotate annually)

#### 6. Monitoring & Audit
- **Immutable audit logs** (append-only, never modify)
- **Real-time alerts** (unusual activity detected)
- **Forensic capability** (reconstruct what happened)
- **Correlation IDs** (trace across services)

## 🔑 Secrets Management & Rotation

**Problem**: Secrets (API keys, passwords) are attack targets.

**Solution**: Store securely, rotate automatically, respond immediately to compromise.

### Secrets Vault

**ALWAYS**:
- Store in AWS Secrets Manager, Azure Key Vault, or HashiCorp Vault
- Never commit to git (use .gitignore)
- Never hardcode (inject at runtime)
- Encrypt at rest and in transit

**Example** (AWS Secrets Manager):
```csharp
public class SecretService
{
    private readonly IAmazonSecretsManager _secretsManager;
    
    public async Task<string> GetSecretAsync(string secretName)
    {
        var response = await _secretsManager.GetSecretValueAsync(
            new GetSecretValueRequest { SecretId = secretName }
        );
        return response.SecretString;  // Database password, API key, etc.
    }
}

// Usage
var dbPassword = await _secretService.GetSecretAsync("prod/db/password");
// Never: var password = "hardcoded_123";
```

### Automated Rotation

**Every 30-90 days**, rotate secrets:

```python
# AWS Lambda - Rotate RDS password
def lambda_handler(event, context):
    secret_name = event['SecretId']
    secrets = boto3.client('secretsmanager')
    rds = boto3.client('rds')
    
    # Get current secret
    current = secrets.get_secret_value(SecretId=secret_name)
    
    # Generate new password
    new_password = secrets.generate_random_string(32)
    
    # Update RDS
    rds.modify_db_instance(
        DBInstanceIdentifier='prod-db',
        MasterUserPassword=new_password,
        ApplyImmediately=False
    )
    
    # Store new version
    secrets.put_secret_value(
        SecretId=secret_name,
        SecretString=new_password
    )
```

### Compromise Response

**If secret is leaked**:
1. **Immediately revoke** (disable the secret)
2. **Generate new** (create replacement)
3. **Update all references** (alert dependent services)
4. **Investigate** (how was it leaked?)
5. **Alert security team** (forensics, regulatory)

## 📋 Compliance Frameworks

### SOC 2 Type II

**What**: Security controls audit over 12 months (not snapshot).

**Who needs it**: Cloud services, SaaS companies (customers require it).

**Key Controls**:
- Security (C1-C2): risk assessment, policies, event logging, access controls
- Availability (A1): uptime, disaster recovery
- Processing Integrity (PI1): accuracy of processing
- Confidentiality (C1): data protection, encryption
- Privacy (P1-P8): GDPR compliance, data handling

**Example Evidence**:
```
- User access reviews (quarterly) ✅
- Configuration baselines (documented) ✅
- Incident logs (searchable, 1+ year) ✅
- Change management approvals (tracked) ✅
- Penetration test results (annual) ✅
- Disaster recovery drills (semi-annual) ✅
```

### ISO 27001

**What**: Information Security Management System (ISMS) certification.

**Who needs it**: Critical infrastructure, regulated industries.

**Key Categories**:
- A.5: Organizational controls (10)
- A.6: People controls (7) — screening, training, awareness
- A.7: Asset management (8) — inventory, classification
- A.8: Access control (11) — user registration, PAM
- A.9: Cryptography (2) — encryption, key management
- A.10: Physical & environmental (8) — perimeter, access
- A.11: Operations (13) — backup, malware, incident mgmt
- A.12: Communications (7) — network security, TLS
- A.13: System acquisition (6) — secure SDLC, testing
- A.14: Supplier relations (3) — agreements, monitoring

### HIPAA (Healthcare)

**What**: Healthcare data protection (U.S.).

**Key Rules**:
- **Privacy Rule** — What data can be used/shared
- **Security Rule** — Technical safeguards (encryption, access control)
- **Breach Notification** — 60-day notification requirement
- **Business Associate Agreements** — Vendor requirements

**Example**:
```csharp
// HIPAA-compliant patient data access
public async Task<PatientRecord> GetPatientDataAsync(string patientId, string userId)
{
    // 1. Verify authorization (minimum necessary principle)
    var access = await CheckAccessAsync(userId, patientId);
    if (!access.IsAuthorized) throw new UnauthorizedException();
    
    // 2. Log all access (audit trail, 6 years)
    await AuditLog.LogAsync(new
    {
        PatientId = patientId,
        AccessedBy = userId,
        AccessTime = DateTime.UtcNow,
        AccessLevel = access.Level
    });
    
    // 3. Retrieve encrypted data (AES-256)
    var encrypted = await db.GetPatientDataAsync(patientId);
    var decrypted = Decrypt(encrypted);
    
    // 4. Apply minimum necessary redaction
    return RedactByAccessLevel(decrypted, access.Level);
}
```

### PCI-DSS (Payment Cards)

**What**: Secure credit card processing.

**Key Requirements**:
- Build **Cardholder Data Environment (CDE)** (isolated)
- **Tokenize cards** (never store PAN, PIN, magnetic stripe)
- **Encrypt in transit** (TLS 1.2+) and at rest (AES-256)
- **Network segmentation** (CDE isolated from internet)
- **MFA for admin** access
- **Annual penetration tests** & scanning

**Example**:
```csharp
// PCI-DSS compliant payment processing
public async Task<PaymentResult> ProcessPaymentAsync(CardData card, decimal amount)
{
    // NEVER store full card
    // ALWAYS tokenize immediately
    var token = await _tokenizationService.TokenizeAsync(card);
    
    // Process with token (card never stored)
    var result = await _paymentGateway.ChargeAsync(
        token,  // Safe to store
        amount,
        idempotencyKey: Guid.NewGuid().ToString()
    );
    
    // Store only: token, amount, status
    // NEVER store: PAN, PIN, CVV, magnetic stripe
}
```

## 🧪 Penetration Testing

**Goal**: Find vulnerabilities before attackers do.

### Test Planning

**Before** starting:
1. **Define scope** (systems in-scope, out-of-scope)
2. **Get authorization** (signed Rules of Engagement)
3. **Set test windows** (off-peak, avoid production)
4. **Establish escalation** (emergency contacts)

### NIST Phases

1. **Reconnaissance** — Gather information (WHOIS, DNS, social engineering)
2. **Scanning** — Find live systems (nmap, service enumeration)
3. **Enumeration** — Extract details (versions, APIs, directories)
4. **Exploitation** — Attempt compromise (OWASP Top 10, auth bypass)
5. **Post-exploitation** — Determine access level
6. **Reporting** — Document findings (CVSS scores, remediation)

### OWASP Top 10

**Test for these vulnerabilities**:
1. Broken access control
2. Cryptographic failures
3. Injection (SQL, command)
4. Insecure design
5. Security misconfiguration
6. Vulnerable components
7. Authentication failures
8. Data integrity failures
9. Logging/monitoring failures
10. Server-side request forgery (SSRF)

### CVSS Scoring

**Example**: SQL injection vulnerability
```
CVSS 3.1 Score: 9.8 (CRITICAL)
- Attack Vector: Network (exploitable remotely)
- Attack Complexity: Low (no special conditions)
- Privileges Required: None (unauthenticated)
- User Interaction: None (automatic)
- Confidentiality Impact: High (all data accessible)
- Integrity Impact: High (all data modifiable)
- Availability Impact: High (database down)
```

### Remediation Timeline

- **Critical**: 24-48 hours
- **High**: 1 week
- **Medium**: 2 weeks
- **Low**: 1 month

## 🚨 Incident Response

**Goal**: Respond quickly, recover, learn, improve.

### IR Playbook

**Severity 1** (system outage, data breach):
- Response time: 15 minutes
- Escalation: CEO, CISO, Legal
- Communication: Every 30 minutes (internal + customer)

**Severity 2** (service degradation):
- Response time: 1 hour
- Escalation: VP Engineering
- Communication: Every 2 hours

**Severity 3** (minor issues):
- Response time: 4 hours
- Escalation: Team lead
- Communication: Daily

### War Room Setup

1. **Declare incident** (severity, incident commander assigned)
2. **Activate war room** (Slack + Zoom)
3. **Investigate** (root cause, preserve evidence)
4. **Mitigate** (temporary fix to restore service)
5. **Recover** (permanent fix)
6. **Communicate** (status updates, no surprises)
7. **Review** (post-mortem within 48 hours)

### Blameless Post-Mortem

**Goal**: Improve systems, not punish people.

**Structure**:
1. **Timeline** (what happened, when)
2. **Root cause** (5 whys analysis)
3. **Contributing factors** (system weaknesses)
4. **Action items** (fix with owners & deadlines)
5. **Lessons learned** (what will we remember)
6. **Follow-up** (verify fixes implemented)

**Example**:
```markdown
# Post-Mortem: Database Connection Failures (30 min outage)

## Timeline
08:15 - Alert: DB connections failing (5% error rate)
08:22 - Incident declared (Severity 1)
08:35 - Root cause: SSL certificate expired
08:40 - Certificate renewed
08:45 - Connections restored

## Root Cause (5 Whys)
1. Why: Certificate expired at 08:15
2. Why: Renewal script failed silently
3. Why: No error logging for script failures
4. Why: Certificate renewal not automated
5. Why: Manual process, no monitoring

## Action Items
- [ ] Automate cert renewal (Lambda function) — Owner: Carol — Due: 2026-04-16
- [ ] Alert on cert expiration (7 days before) — Owner: Bob — Due: 2026-04-13
- [ ] Test renewal monthly — Owner: Carol — Due: 2026-04-20

## Lessons Learned
- "Never assume 'automatic' renewal works without monitoring"
- "Silent failures hide problems"
- "Monitoring metrics ≠ alerts"
```
